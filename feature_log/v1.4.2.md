# FlightScanner v1.4.2 补丁更新

**发布日期**: 2026-04-30

**版本**: v1.4.2

---

## 版本概述

本次补丁聚焦两条主线：

1. **去哪儿国内线采集从根本上打通** —— Qunar 对国内线 `touchInnerList` 响应做了反爬数据混淆（scrambled JSON 片段 + 内嵌 IIFE，浏览器侧 eval 后才还原），Python 端解析永远是乱码。改为从 React 渲染完成的 DOM 里直接抓数据，绕开整套反爬。
2. **往返程采集管道全链路修复** —— 国内往返首次落地，连带把 top-N 截断、路线过滤、batch_id 继承、回程时间窗四个长期隐性 bug 一并解决。

这些修复本身对所有用户都有益（不依赖 v1.5.0 的周末雷达），所以单独成版。

---

## 修复 1：去哪儿国内线 DOM 抓取

### 问题

桌面端 `wwwInnerList` / `wbdflightlist` 走得通时（国际线大多数场景）一切正常，但国内线下：

- 桌面 `.b-airfly` 选择器 20s 超时
- `wbdflightlist` 返回 `ret:false, code:-4 请求必须参数中存在空值`
- 移动端 `touchInnerList` **抓得到**响应（3MB）但解析失败：`data` 字段不是合法 JSON，前 2000 字符以 `ailItemTitle"` 开头（非 `{`/`[`），后 2000 字符是 `_0x96f9b1['data']=JSON['parse'](_0x3c414c['join'](''))...` 的混淆 JS

### 根本原因

去哪儿对国内线启用了**响应数据混淆**：

```
服务端返回： {"ret":true,"data":"<scrambled JSON 片段>...<内嵌混淆 JS>"}
浏览器：      eval(data) 读取 <meta name="meta-pro" content="..."> 作密钥，
             调用 array.shift/push 洗牌 + join('') 还原真 JSON，再 JSON.parse
客户端只能拿到还原后的结果；Python 拿到的永远是乱序字节
```

本地无 Node.js/deno/quickjs，无法直接 eval；就算能 eval，那段 JS 也依赖页面 DOM（meta 标签），脱离浏览器执行不了。

### 修复：改从 DOM 取数

浏览器 React 应用渲染完后，真实航班数据落在 `<ul class="list-content"><li class="list-row item">` 里。2026-04 实测 DOM 结构：

```html
<li class="list-row item">
  <div class="list-info">
    <div class="airpot-info">
      <div class="from-info"><p class="from-time">07:55</p><p class="from-place">浦东T2</p></div>
      <div class="time-info"><p class="howlong"><span class="time">2时30分</span></p>
                             <p><span class="stop-city"></span></p></div>  <!-- 空=直飞 -->
      <div class="to-info"><span class="add-day"></span>  <!-- +1天 -->
                           <p class="to-time">10:25</p><p class="to-place">白云T1</p></div>
    </div>
    <div class="company-info"><span class="company1">国航CA8327 空客321(中)</span></div>
  </div>
  <div class="price1"><p class="price-info"><span class="i-yen"></span><span>420</span></p></div>
</li>
```

**新增 [`_parse_mobile_dom()`](../src/flightscanner/scrapers/qunar_scraper.py)**：

- `wait_for_selector("ul.list-content li.list-row.item", timeout=20s)` 等列表渲染
- `re.search(r"[A-Z0-9]{2}\d{3,4}")` 从 `.company1` 文本抽航班号，前缀作为航司名
- 代号共享行 `东航MU5307\n南航CZ8519` 取第一行（实际承运方）
- 中转航班（`.stop-city` 非空）直接跳过
- `.add-day` 非空时联合时刻计算跨日到达日期，冲突以 `+N天` 标注为准

**`_search_via_mobile_api()` 流程调整**：

1. 保留 `page.route("**/touchInnerList**")` 拦截作为国际线兜底
2. 若 3 秒后已有拦截响应但 `_has_valid_response()` 一直 False（反爬混淆的标志），**提前退出轮询**，省 27 秒无谓等待
3. 进入 DOM 解析；成功直接返回，失败才回退 API 解析路径

源 tag 统一为 `qunar`（原 `qunar_api` / `qunar_mobile` 归并，避免 dedup 因 source 不同重复保留同一航班）。存量 DB 记录已回填。

## 修复 2：国内往返程分程采集

### 问题

`search_flights(params)` 的往返分支里只有 `is_international_route()` 的处理；国内线带 `return_date` 时直接掉进单程搜索路径，返回的 154 条全是去程，上游 `_combine_roundtrip_prices` 拿不到回程，告警 `往返程配对失败：去程 N 条，回程 0 条`。

### 修复

在 `search_flights()` 的国际往返 if 后增加国内往返分支，复用已有的 [`_search_inter_roundtrip_fallback`](../src/flightscanner/scrapers/qunar_scraper.py) 分程搜索：

```
search_flights(上海↔广州, dep, ret)
  └─ _search_inter_roundtrip_fallback
        ├─ search_flights(上海→广州, dep)              # 去程，方向 DEPARTURE
        └─ search_flights(广州→上海, ret)              # 回程，方向 RETURN
              └─ (自动改方向标记为 RETURN)
```

helper 文案通用化：日志前缀 `[往返降级]` → `[分程往返]`，docstring 同时覆盖国际/国内两种场景。

## 修复 3：方向感知 top-N 截断

### 问题

分程往返返回 40 条单程记录（20 DEPARTURE + 20 RETURN），[`_scrape_oneway`](../src/flightscanner/scheduler/price_monitor.py) 的 `保留最低 N 条` 把 40 条**混在一起**按价格排序取前 20。去程和回程票价接近，top 20 容易偏向其中一侧，另一侧可能 0 条 → `_combine_roundtrip_prices` 无法配对。

### 修复

按方向分别排序切 top N，拼接返回：

```python
departures = sorted([fp for fp in result if fp.flight_info.direction == DEPARTURE], key=price)[:N]
returns    = sorted([fp for fp in result if fp.flight_info.direction == RETURN],    key=price)[:N]
top = departures + returns    # 最多 2N 条，下游合成后约 N 条组合记录
```

日志区分两种场景：
- 单程：`QunarScraper 采集到 X 条结果，保留最低 N 条`（行为不变）
- 往返：`QunarScraper 采集到 X 条结果（去程 A / 回程 B），保留最低 A+B 条（去程+回程各最多 N）`

## 修复 4：路线过滤器前置

### 问题

用户给路线配置了 `dep_time_from='19:00'`（只看晚班）后，Qunar 采到 154 条却 0 条进入最终结果——全是 Ctrip 数据。

排查：管道顺序是 `采集 → per-platform 切 top N → 过滤器 → 存库`。Qunar 的 top 20 最便宜里多是清晨/凌晨特价票，过滤器按 `dep_time >= 19:00` 一扫全清光。Ctrip 的 top 20 含一些晚班直接过滤通过。

### 修复

把路线过滤前置到截断**之前**：「过滤合格集合里最便宜的 N 条」。`_scrape_all_platforms` / `_scrape_oneway` 新增可选 `route` 参数：

```python
result = await scraper.search_flights(params)        # 原始 154 条
if route is not None:
    result = self._apply_route_filters(route, result)  # 154 → 42 条合规
top = sorted(...)[:N]                                # 取合规集的最便宜 20 条
```

截断后 `_combine_roundtrip_prices` → `_apply_route_filters`（幂等再筛一次，组合记录需要两段都合规）→ 存库。

## 修复 5：回程时间窗过滤

### 问题

`_apply_route_filters` 原本只有 `dep_time_from/to`、`arr_time_from/to` 四个时间窗字段，全部按出发机场/时间语义。对分程往返里的 RETURN 方向单程记录（广州→上海），用去程时间窗去筛是错的。

### 修复

**Schema** ([`models/database.py`](../src/flightscanner/models/database.py))：`routes` 表新增 4 列 + 幂等迁移：

```sql
ret_dep_time_from TEXT    -- 回程起飞窗口开始
ret_dep_time_to   TEXT    -- 回程起飞窗口结束
ret_arr_time_from TEXT    -- 回程落地窗口开始
ret_arr_time_to   TEXT    -- 回程落地窗口结束
```

`RouteWithLatestPrice` / `add_route()` / `get_all_routes()` 全链路透传。

**过滤器重写** ([`_apply_route_filters`](../src/flightscanner/scheduler/price_monitor.py))：按记录类型分流——

| 记录类型 | 去程段校验 | 回程段校验 |
|---|---|---|
| 组合记录 (`return_flight_info` 非空) | `dep_*` + `arr_*` | `ret_dep_*` + `ret_arr_*` + **机场反向** |
| 单程 RETURN（分程中间态） | — | `ret_dep_*` + `ret_arr_*` + 机场反向 |
| 单程 DEPARTURE / 无方向 | `dep_*` + `arr_*` | — |

机场反向：回程段的 `departure_airport_code` 应匹配路线的 `arr_airport_code`（回程起点 = 目的地），反之亦然。

路线未配置回程窗口时，RETURN 记录全部放行（向后兼容）。

## 修复 6：组合记录继承 `batch_id`

### 问题

日志显示 `已保存 15 条`，实际查 DB 同批次只有 8 条 Ctrip 记录，7 条 Qunar 的 `batch_id=None`。UI 的"最新价格"查询硬性要求 `batch_id IS NOT NULL`（route_service.py），Qunar 数据被静默过滤。

### 根本原因

调度器 `scrape_route()` 顺序：

```python
1. flight_prices = _scrape_all_platforms(...)    # 20 outbound + 20 return 单程 + Ctrip 20 组合
2. for fp in flight_prices: fp.batch_id = batch_id   # 打 batch_id ✓
3. flight_prices = _combine_roundtrip_prices(...)    # ← 配对时 new FlightPrice(...)
4. 存库
```

步骤 3 的 `_combine_roundtrip_prices` 把去程+回程配对成新组合记录时，`FlightPrice(...)` **漏传了 `batch_id`**，默认 None。

### 修复

```python
newly_combined.append(FlightPrice(
    ...,
    batch_id=out_fp.batch_id,   # 继承去程的批次 ID
))
```

存量 34 条 `batch_id=None` 的历史 qunar_api 记录已按 `(route_id, source, 时间簇)` 分组回填合成的 batch_id。

## 修复 7：前端来源映射前缀容错

### 问题

表格和图表的来源列本应显示「去哪儿」/「携程」，实际出现原始字符串 `qunar_api`。

### 根本原因

[`charts.py`](../ui/components/charts.py) 的 `_SOURCE_LABELS = {"qunar": "去哪儿", "ctrip": "携程"}` 是精确匹配，遇到 `qunar_api` / `qunar_mobile` fallthrough 到原串。

### 修复

双层兜底：

1. **源头统一**：爬虫内 4 处 `source="qunar_api"` / `source="qunar_mobile"` 全改为 `source="qunar"`；存量记录 `UPDATE price_history SET source='qunar' WHERE source IN ('qunar_api','qunar_mobile')`（25 行）
2. **UI 前缀匹配**：`_source_label()` 先查精确映射，未命中再前缀匹配，任何 `qunar_*` 变体都能显示「去哪儿」
