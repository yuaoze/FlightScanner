# FlightScanner v1.0.3 修复更新

**发布日期**: 2026-03-17

**版本**: v1.0.3

## 版本概述

v1.0.3 为纯 bug 修复版本，集中解决三类问题：

1. **最低价计算错误**：同一采集批次的价格因微秒时间差被错误分组，导致显示错误最低价
2. **Qunar 国际/跨境路由 0 结果**：涉及非中国城市的路由使用了错误的 URL 格式与页面
3. **SQLite 并发写入异常**：多线程共享 engine 未配置 `check_same_thread=False`

---

## Bug 1：最低价计算错误（核心）

### 问题复现

**现象**：路线卡片标题、平台价格块、趋势图最新点显示不一致。
例如去哪儿平台价格块显示 ¥1563，而实际采集的最低价应为 ¥303。

### 根本原因

同一次采集会话中，Playwright 异步写入导致各条记录的 `scraped_at` 有微秒级差异：

```
03:17:50.769902 → ¥303   ← 实际最低价
03:17:50.848727 → ¥324
03:17:50.891370 → ¥386
03:17:50.961887 → ¥1563
03:17:50.988060 → ¥1563
03:17:51.000917 → ¥1563  ← MAX(scraped_at) 命中此处
```

旧查询逻辑：`MIN(price) WHERE scraped_at = MAX(scraped_at)`
→ 只命中最后 1 条（或同微秒的几条），MIN 结果为 ¥1563

正确结果：应对整批次 7 条记录取 MIN，结果为 ¥303

前端 `_render_source_price_summary`（overview.py）也使用精确时间戳匹配，同根同源。

### 修复方案：batch_id 批次标记

在所有记录中增加 `batch_id` 字段，唯一标识一次采集会话，以此替代精确时间戳分组。

**batch_id 格式**：`route_{route_id}_{ISO时间戳}_{MD5前8位}`
**示例**：`route_5_2026-03-17T11:17:50.123456+00:00_abc12345`

#### 修改的文件

**`src/flightscanner/interfaces.py`**

`FlightPrice` dataclass 新增字段：
```python
batch_id: Optional[str] = None
```

**`src/flightscanner/models/database.py`**

`PriceHistory` 模型新增列及索引：
```python
batch_id = Column(String(100), nullable=True, index=True)
# 复合索引
ix_price_history_route_batch: (route_id, batch_id)
ix_price_history_source_batch: (source, batch_id)
```

`_apply_migrations()` 补充迁移语句（兼容已有数据库）：
```python
"ALTER TABLE price_history ADD COLUMN batch_id TEXT"
```

**`src/flightscanner/scheduler/price_monitor.py`**

`scrape_route()` 在存库前统一生成 batch_id 并标记所有记录：
```python
batch_timestamp = datetime.now(timezone.utc).isoformat()
hash_input = f"{route.id}_{''.join(fp.source for fp in flight_prices)}_{batch_timestamp}"
batch_hash = hashlib.md5(hash_input.encode()).hexdigest()[:8]
batch_id = f"route_{route.id}_{batch_timestamp}_{batch_hash}"
for fp in flight_prices:
    fp.batch_id = batch_id
```

**`src/flightscanner/core/services/route_service.py`**

`get_all_routes()` 子查询重写：

旧逻辑（错误）：
```python
# 按平台找最新 scraped_at，再匹配精确时间
func.max(PriceHistory.scraped_at).label("latest_scraped_at")
PriceHistory.scraped_at == latest_per_source_subq.c.latest_scraped_at
```

新逻辑（正确）：
```python
# 按平台找最新 batch_id，再匹配整批次
func.max(PriceHistory.batch_id).label("latest_batch_id")
PriceHistory.batch_id == latest_batch_per_source_subq.c.latest_batch_id
```

`save_price_for_route()` 写入 batch_id；`get_route_price_history()` 返回含 batch_id 的 FlightPrice。

**`ui/components/overview.py`**

`_render_source_price_summary()` 改为按 batch_id 分组取最低价：
```python
# 旧逻辑
if fp.scraped_at == latest_time.get(src):  # 精确时间戳匹配

# 新逻辑
if bid != latest_batch.get(src):           # 批次 ID 匹配
```
保留旧数据（无 batch_id）的时间戳兜底逻辑。

## Bug 2：Qunar 国际/跨境航线搜索 0 结果

### 问题复现

**现象**：MNL→CEB（马尼拉→宿雾）等非中国城市路由返回 0 条结果。

### 根本原因

旧代码对国际路由使用了错误的 URL 格式：

```python
# 旧逻辑（错误）：国际用 fromCity/fromDate/isInter=true
base_url = "https://flight.qunar.com/site/oneway_list.htm"
url = f"{base_url}?fromCity={city}&fromDate={date}&isInter=true"
```

Qunar 实际页面区分：
| 场景 | 页面 |
|---|---|
| 两端均为中国城市 | `oneway_list.htm` / `roundtrip_list_new.htm` |
| 至少一端不是中国城市 | `oneway_list_inter.htm` / `roundtrip_list_inter.htm` |

所有情况均使用 `searchDepartureAirport` / `searchArrivalAirport` 参数格式。

### 修复

**`src/flightscanner/scrapers/qunar_scraper.py`**

删除国际/国内 URL 格式分支，统一参数格式，只切换页面路径：

```python
if params.return_date:
    base_url = (
        "https://flight.qunar.com/site/roundtrip_list_inter.htm" if intl
        else "https://flight.qunar.com/site/roundtrip_list_new.htm"
    )
else:
    base_url = (
        "https://flight.qunar.com/site/oneway_list_inter.htm" if intl
        else "https://flight.qunar.com/site/oneway_list.htm"
    )
# URL 参数格式与国内完全相同（searchDepartureAirport 等）
```

**`src/flightscanner/utils/city_codes.py`**

更新 `is_international_route()` 注释，明确其语义是"是否需要 `_inter` 页面"，而非"是否真正的国际航班"。

## Bug 3：SQLite 并发写入异常

### 问题复现

**现象**：爬虫采集时报 `sqlite3.OperationalError: attempt to write a readonly database`。

### 根本原因

多个问题叠加：

1. **多线程 engine 共享**：APScheduler 采集线程与 Streamlit 主线程共享同一 `engine`，SQLite 默认 `check_same_thread=True` 阻止跨线程连接使用
2. **WAL 模式未开启**：默认 journal 模式下读写互斥，Streamlit 持有读连接时爬虫写入被阻塞
3. **僵尸进程**：多个 Streamlit 进程（PID 71666 等）持有旧数据库文件描述符未释放

### 修复

**`src/flightscanner/models/database.py`**

```python
# 新增 check_same_thread=False + WAL 模式
connect_args = {"check_same_thread": False} if db_url.startswith("sqlite") else {}
engine = create_engine(db_url, echo=False, pool_pre_ping=True, connect_args=connect_args)

if db_url.startswith("sqlite"):
    with engine.connect() as conn:
        conn.execute(text("PRAGMA journal_mode=WAL"))
        conn.commit()
```

**`.gitignore`**

补充 SQLite WAL 附属文件：
```
*.db-shm
*.db-wal
```

## Bug 4：Ctrip 往返记录被误当单程处理

### 问题复现

**现象**：携程往返搜索结果在 `_combine_roundtrip_prices()` 中被误当单程去程记录处理，导致往返价格丢失。

### 根本原因

携程 API 返回的往返记录（`adultPrice` = 两程合计价格）没有设置 `return_flight_info`，下游配对逻辑无法识别其为已合并的往返记录。

### 修复

**`src/flightscanner/scrapers/ctrip_scraper.py`**

往返搜索时为每条记录创建虚拟回程占位符，防止被 `_combine_roundtrip_prices()` 误处理：
```python
if params.return_date is not None:
    return_flight_info = FlightInfo(
        flight_no="VIRTUAL_RETURN",
        departure_city=params.arrival_city,
        arrival_city=params.departure_city,
        departure_date=params.return_date,
        direction=FlightDirection.RETURN,
        ...
    )
```

同时新增座位可用性过滤：跳过 `seatsLeft=0` 或 `seatsLeft=None` 的已售罄记录。

## 升级说明

本版本包含数据库 schema 变更（新增 `batch_id` 列）。

**新数据库**：`init_db()` 通过 `create_all()` 自动建表，无需手动操作。

**已有数据库**：`_apply_migrations()` 在启动时幂等执行 `ALTER TABLE`，自动添加 `batch_id` 列，兼容旧数据（旧记录 `batch_id=NULL`，前端有兜底逻辑）。

```bash
# 重启即可（迁移自动执行）
PYTHONPATH=/path/to/FlightScanner venv/bin/streamlit run ui/app.py
```
